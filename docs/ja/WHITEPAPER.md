## 0. エグゼクティブ・サマリー

### 何を作るか

MORM（モルム）は、**TikTok級のショート動画体験**と**詐欺不能なP2Pコマース**を、**完全分散型インフラ**の上に統合する。既存のSNS、決済、ストレージ、CDN、Eコマースの一部を置き換える「Webスタック全体の代替物」である。

### なぜ今か（Why Now）

3つの巨大な転換点が2026年に重なる：

1. **クリエイターエコノミーの崩壊** — 30%手数料、突然のシャドウバン、AIによる無断学習への不満がピークに達した
2. **Web3 UXの成熟** — Passkey/FIDO2、ERC-4337、C2PA等の標準化により、シードフレーズなしでブロックチェーンを使える時代がついに来た
3. **AI生成物の氾濫** — 真贋証明への社会的需要が決定的になり、各国規制（EU AI Act等）が動き始めた

### 市場規模（TAM）

|---|---|---|
| ショート動画SNS | 約 $3,000億/年 | Statista 2026予測 |

### 立ち位置の正直な宣言

MORMは**プリミティブの発明者ではない**。本書は誠実な姿勢で、各構成要素について先行プロジェクトを認識した上で、「初の統合スタック」としての価値を主張する。

真のオリジナリティ・モートは以下の3点に絞られる：

1. **Proof of Physical Evidence（PoPE）** — ブロックハッシュ透かし入り梱包/開封動画によるエスクロー解放（**商用シップ初**）
2. **Slash + 生体認証永久ban** — 不正検知時の財務没収と生体ベース永久排除の結合（**前例なし**）
3. **動画特化統合PoUW L1** — トランスコード+AI推論+配信を単一合意基盤にした初の動画L1

### 運営構造の宣言

MORMには**法人が存在しない**。Corporation も Foundation も DAO LLC もない。代わりに：

- **Smart Contract**（不変の資金フロー）
- **AI Agent群**（日常運営の自律実行）
- **DAO**（パラメータ決定）

の3層で完全自律運営される。詳細は [AGENTS.md](AGENTS.md)。

## 1. 課題（Problem Statement）

### 1.1 中央集権プラットフォーマーの限界

| 課題 | 既存プラットフォームの状況 |
|---|---|
| 不透明な検閲 | TikTokシャドウバン、YouTubeのDMCA濫用 |
| 高額手数料 | App Store/Google Play 30%、Meta/TikTok 50%超 |
| ポリシーの突然変更 | YouTube収益化条件の度重なる変更 |
| データ主権の喪失 | 個人データの企業所有、AI学習への無償提供 |

### 1.2 既存分散型サービスの限界

「Web3で代替する」という構想は10年前から繰り返されてきた。なぜ普及しなかったか：

- **遅延** — IPFSベース配信は数秒〜数十秒のロード時間
- **複雑さ** — ウォレット管理・ガス代・チェーン選択の認知負荷
- **品質** — スパム・コピー・低品質コンテンツの氾濫
- **規模** — ネットワーク効果が中央集権に勝てなかった

### 1.3 P2Pコマースの限界

- OpenBazaarは2018年に活動停止（信頼問題）
- Multisigエスクローだけでは詐欺を防げない
- 既存P2Pマーケットも仲介業者依存で10〜30%手数料

**MORMはこれらすべてに、既存技術の精密な統合と少数のオリジナル設計で応える。**

### 3.2 真のオリジナリティ・モート

#### Moat 1: Proof of Physical Evidence（PoPE）

商用シップ済みのP2Pコマースで、ブロックハッシュ透かし入り動画をエスクロー解放条件とするものは存在しない。

OpenBazaar/Particl/OriginはMultisigエスクローのみ。Princeton大学の2018年論文は概念実証に留まり、製品化されなかった。MORMは初の商用実装となる。

#### Moat 2: Slash + 生体認証永久ban

Worldcoin/Humanity ProtocolはSybil防止のみで、不正者を網羅的に排除する仕組みではない。MORMは:

- 財務的Slash（ステーク没収）
- 生体認証ベースのネットワーク永久排除
- 別デバイス・別ウォレットでの再参加を技術的に不可能化

を結合する。法的設計（GDPR/APPI/CCPAの同意取得）込みで実装する。

#### Moat 3: 動画特化統合PoUW L1

Livepeer（トランスコード+AI）、Theta（配信）、Filecoin（ストレージ）が分離している領域を、単一のPoUW合意基盤に統合する初の動画L1。

## 4. システム・アーキテクチャ

### 4.1 ネットワーク階層

| Tier | 名称 | リソース | 役割 |
|---|---|---|---|
| Tier 1 | **Validator (NodePower)** | GPU, CPU, RAM, NVMe | エンコード、AI解析、ブロック生成、検証 |
| Tier 2 | **Edge Node** | SSD, 広帯域 | MORM Cells保持、近隣デバイスへの配信 |
| Tier 3 | **Light Client** | スマホ／PC／ゲーム機 | 視聴、UI、一時キャッシュ中継 |

すべてのTierがP2Pネットワークの一部。Light Clientも余剰リソース提供時にはEdge Node的に振る舞う。

### 4.2 通信層

- **QUIC（UDPベース）** を全層共通で採用
- ブラウザ層では **WebAssembly + WebTransport** でP2P参加を実現
- libp2pをベースとし、3秒セグメントの並列ストリーミングに最適化

### 4.3 MORM Chain

| 項目 | 仕様 |
|---|---|
| 構造 | DAGベース並列ブロック生成 |
| 合意 | Proof of Useful Work (PoUW) |
| 確定性 | 数秒以内の即時確定 |
| 設計 | ステートレス（Light Clientは履歴を持たない） |
| トランザクション・スループット | 設計目標 100,000 TPS |

PoUWは「動画トランスコード」「AI解析」「配信」「物理エビデンス検証」を有用な仕事として証明し、報酬と合意を一体化する。

## 5. ストリーミング・プロトコル（MORM Cells）

### 5.1 セグメント仕様

| 項目 | 値 |
|---|---|
| 形式 | WebM (VP9 / AV1) |
| 長さ | 3秒 |
| フレームレート | ≤ 30 fps |
| サイズ目安 | 数百KB 〜 1MB / セグメント |

### 5.2 50%/10%キャッシュサイクル

```
[再生開始]
   ↓
0.3秒（10%） → 次のCellをBG予約
   ↓
1.5秒（50%） → 古いCellをメモリからパージ
   ↓
3.0秒（100%） → 次のCellへシームレス接続
```

メモリ消費はフラット維持、再生は途切れない。

### 5.3 先行技術との関係

3秒セグメントとプリフェッチ自体はHLS/DASH（標準2-10秒）、Akamai Segment Prefetch、LL-HLSの`EXT-X-PRELOAD-HINT`等で確立済み。MORMの貢献は：

- パラメータ最適化（TikTok級の体感を分散環境で再現）
- PoUWとの連動（プリフェッチ作業もNodePowerが報酬対象）
- WebMネイティブ（ほとんどの分散型動画はHLS/MP4）

### 5.4 ロード時間のタスク化

分散ネットワーク特有の初動遅延を、UXで吸収する：

- インタラクティブ広告（タップでトークン獲得）
- AIアノテーション・タスク（タグ付け・カテゴリ検証）
- 予測市場ミニゲーム（次の動画のバズ予想）

## 6. ID・認証システム（Walletless）

### 6.1 設計思想

シードフレーズ・秘密鍵管理を完全に隠蔽する。これは2025-2026時点で**業界標準**となった方向性であり、MORMはこれを**生体ban機構と統合**することで独自性を出す。

### 6.2 技術スタック

- **Passkeys（FIDO2）**: FaceID / TouchID / Windows Hello / ゲーム機の生体認証
- **Secure Element**: 各デバイスのセキュアチップ内に専用鍵を生成
- **MPC的鍵分散**: デバイスチップとネットワーク断片に分割
- **Social Recovery**: 複数所有デバイスを使った復旧

### 6.3 先行技術との関係

| プロジェクト | 提供 | MORMの差分 |
|---|---|---|
| Coinbase Smart Wallet | Passkey + ERC-4337 | 動画SNSと統合、生体banと連動 |
| Privy | Embedded wallet + Passkey | 同上 |
| Web3Auth | MPC + Social login | 同上 |

MORMは「ID自体の発明」ではなく、「IDをアプリ全体（投稿権限、Slash、復旧）と密に統合する設計」を提供する。

## 7. コンテンツ純度プロトコル

### 7.1 V-Hash（重複検知）

- 知覚ハッシュ（pHash）で映像特徴量
- オーディオ・フィンガープリントで音声波形
- ベクトルDBで類似度検索

**先行技術**: YouTube Content ID（2007〜）、Audible Magic、pHash.org。MORMは標準技術を採用し、独自貢献は**オンチェーン優先順位ルール**にある。

### 7.2 重複処理

- **タイムスタンプ・レース**: 最先のブロック記録を「正」と確定
- **Garbage Collection**: 重複Cellsはネットワーク全体に消去命令
- **アップロード拒否**: 既登録ハッシュは投稿ゲートウェイで弾く

### 7.3 Generation ID（C2PA準拠）

MORM内蔵の生成AI動画には、**C2PA Content Credentials**準拠のGeneration IDを発行し、MORM Chainに刻印する。

| 互換性 | 内容 |
|---|---|
| C2PA 2.x | Adobe CAI、Google SynthID、OpenAI Sora等と相互運用可能 |
| ERC-7053 | Numbers Protocolの動画NFT標準とブリッジ可能 |
| EU AI Act | 透明性義務に対応 |

これにより、MORM外で生成された動画もC2PAマニフェスト経由で来歴を証明でき、エコシステム全体の真贋エコシステムに参加できる。

## 8. MORM Shop / 信頼プロトコル

これがMORMの最強モートである。

### 8.1 Proof of Physical Evidence（PoPE）

```
[発送者]                       [購入者]
   │                             │
   ├─ MORM専用カメラで梱包動画撮影 │
   │  （ブロックハッシュ透かし）   │
   ├─ 動画ハッシュをチェーン記録    │
   ├─ 商品発送                    │
   │                             │
   │                             ├─ 受取
   │                             ├─ MORM専用カメラで開封動画撮影
   │                             │  （ブロックハッシュ透かし）
   │                             └─ 動画ハッシュをチェーン記録
   ↓                             ↓
   ┌──────────────────────────────┐
   │ AI / Validator による整合性検証   │
   │ ・動体解析                       │
   │ ・コンテンツ照合                 │
   │ ・タイムスタンプ・ハッシュ整合性 │
   └──────────────────────────────┘
            ↓                ↓
      [一致] エスクロー解除   [不一致] 自動Refund + Node-Lock
```

### 8.2 改ざん防止メカニズム

- MORM専用カメラ以外の動画は受け付けない
- 動画フレーム内に撮影時点のブロックハッシュを電子透かしで焼き込む
- 動体解析で不自然なカット・ループ・合成を検知
- カメラAPIはSecure Element経由で認証、AI動画は弾く

### 8.3 スマート・エスクロー

- 購入額の99%をスマートコントラクトにロック
- 双方の動画一致 → 発送者へ送金
- 一致しない、または異議申立期間内に異議 → コミュニティ仲裁またはAI判定
- 詐欺確定 → 即座に購入者へ組み戻し

### 8.4 Node-Lock & Slash + 生体永久ban

不正確定時の自動執行：

1. 不正者のノードを凍結
2. ステーキング・保有報酬を没収（被害者補填+ネットワーク再分配）
3. **デバイスID・生体認証を恒久ブラックリスト化**
4. 該当ユーザーのMORM接続を**全デバイスで永久遮断**

#### 法的設計（重要）

GDPR、APPI、CCPAとの整合のため：

- 生体テンプレートは**ハッシュ化**して保管（原データは保持しない）
- 同意プロセスを段階化（軽度違反では生体banに到達しない）
- データ最小化原則に従う
- 「忘れられる権利」と「Sybil防止のための保持」のバランスを段階制裁で吸収

## 13. ガバナンス（法人なし設計）

### 13.0 基本構造

MORMには**法人（Corporation / Foundation / DAO LLC）が存在しない**。中央運営チームも、CEOも、本社もない。代わりに3層構造で運営される：

```
Layer 3: DAO（トークンホルダー） ← パラメータ決定
Layer 2: AI Agent群             ← 日常運営の自律実行
Layer 1: Smart Contract         ← 不変の資金フロー
```

詳細は [AGENTS.md](AGENTS.md) を参照。

### 13.1 段階的自律化

- **MORM Phase 0**: 仮名コア貢献者による起動、初期Validator稼働、Smart Contract初期デプロイ
- **MORM Phase 1**: TGE後、Adminキー全Renounce、AI Agent本番運用開始、DAO投票機能Live
- **MORM Phase 2**: AI Agentのフルカタログ稼働、創業貢献者特権なし、完全DAOガバナンス
- **MORM Phase 3**: 第二世代AI Agent群への進化、創業貢献者は一参加者として活動

### 13.2 不変パラメータ

DAO投票でも変更不可：

- 総発行量 100億 MORM
- 運営手数料 1.0%
- ユーザー所有データの権利
- Walletless ID原則
- PoPE必須化原則

### 13.3 DAO投票対象（Tier別）

**Tier 1（軽微、48時間、Quorum 3%）**:
- ノード報酬の重み付け係数
- スパム判定閾値の微調整

**Tier 2（中規模、7日、Quorum 10%）**:

**Tier 3（重要、14日、Quorum 20%）**:
- 新Smart Contractデプロイ
- AI Agentの追加・削除
- Bridge追加
- 大口Treasury支出（>$1M相当）

### 13.4 緊急対応

DAOプロセスでは間に合わない事態に対し、**Emergency Multi-sig（5/7署名）**でPause可能。署名者は地理的分散、DAO選任。発動後72時間以内にDAO投票で正式承認/取消。
